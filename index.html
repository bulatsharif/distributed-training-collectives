The collectives you see in GPU training

A collective is a matched operation over a process group. Every participating rank must call the same operation with compatible tensor shapes, datatypes, and ordering. In NCCL, the rank is a logical participant in an `ncclComm_t`, not necessarily the same thing as a CUDA device ordinal.

Equivalences

Why all-reduce is often two collectives in disguise

A full all-reduce can be implemented as `ReduceScatter + AllGather`. The reduce-scatter phase makes each rank responsible for one reduced shard. The all-gather phase then shares those reduced shards until everyone has the complete reduced tensor.

This decomposition is why sharded training likes reduce-scatter: FSDP and ZeRO often do not need the full reduced gradient on every rank. They can keep only the shard needed by that rank's optimizer state and avoid materializing the full tensor.

Algorithm families

How libraries move bytes

NCCL chooses algorithms and protocols using message size, topology, transport capabilities, and runtime tuning. The exact choice is version- and system-dependent, but these families explain the behavior most practitioners see in traces.

Bandwidth first

Ring

Ranks form one or more logical rings. Each rank sends to its next neighbor and receives from its previous neighbor. Large tensors are chunked so every link can stay busy.

  • Great for large messages because each rank injects and drains continuously.
  • All-reduce usually appears as reduce-scatter then all-gather.
  • Latency grows with rank count because data advances hop by hop.
Latency first

Tree

Ranks are arranged as reduction and broadcast trees. Data reaches the root in logarithmic depth, then fans back out. Double-tree designs can use complementary trees to increase utilization.

  • Useful for smaller payloads or high-latency inter-node paths.
  • Handles root-oriented collectives naturally.
  • Can underuse bandwidth if only a few links are active at a time.
Topology aware

Hierarchical

Work is split into intra-node and inter-node phases: reduce within a node over NVLink/PCIe, communicate between node leaders over InfiniBand or Ethernet, then distribute inside each node.

  • Reduces pressure on slow network links.
  • Depends heavily on rank placement and NIC affinity.
  • Common in multi-node GPU clusters.
Specialized fabrics

Switch or network assisted

On capable systems, switches or network devices may accelerate multicast, reduction, or collective network paths. NCCL exposes tuning around CollNet, NVLink SHARP, NIC rails, and transport selection.

  • Can cut GPU or host traffic when hardware supports it.
  • Misconfiguration often looks like slow initialization or unexpected fallback.
  • Debug logs are essential for seeing what NCCL selected.

NCCL execution model

What happens between your framework call and the interconnect

1 Process group

Framework code maps workers to ranks and creates one or more communicators.

2 Collective call

Every rank launches the same operation with compatible tensor metadata.

3 Plan selection

NCCL considers topology, message size, channels, transports, and protocol thresholds.

4 CUDA stream enqueue

Communication kernels and network operations are ordered with stream semantics.

5 GPU/NIC movement

Chunks move across NVLink, PCIe, shared memory, sockets, or RDMA-capable network paths.

6 Completion

Work completes when the stream reaches the collective and all ranks satisfy the protocol.

Topology shapes the ring

NCCL is topology-aware. It tries to build rings, trees, and channels that follow the fast links first: NVLink/NVSwitch inside a server, PCIe paths where necessary, and NIC rails for inter-node movement. On multi-node jobs, the same rank order can perform very differently depending on which process lands on which GPU and NIC.

Ownership axes

How data, tensor, pipeline, and expert parallelism relate

Parallelism is a choice about what is repeated and what is split. Real large-model training usually uses a Cartesian product of process groups: data parallel groups replicate training examples, tensor parallel groups split layer math, pipeline groups split depth, and expert groups split sparse MoE capacity.

When it is useful

DP

Data parallel

Replicate the model over different data shards. Dense DDP syncs gradients with all-reduce; sharded DP/FSDP replaces that with reduce-scatter and all-gather.

TP

Tensor parallel

Split matrices and layer activations inside a layer. It reduces per-rank layer memory, but introduces collectives inside the forward and backward passes.

PP

Pipeline parallel

Split consecutive layers into stages. Microbatches flow through stages, trading activation traffic and scheduling bubbles for lower per-rank model memory.

EP

Expert parallel

Split MoE experts across ranks. Router-selected tokens move to their expert owners, usually with all-to-all style dispatch and combine phases.

EDP

Expert data parallel

Replicate expert-parallel groups across data-parallel replicas. Matching experts across replicas synchronize their gradients, while tokens only route inside each expert group.

CP/SP

Context and sequence axes

Split sequence-length or selected activation dimensions. These axes complement TP and are useful when long contexts make activations, attention, or KV-like state dominate memory.

How to read a hybrid layout

A rank can be in one group for each axis at the same time. For example, `TP=2, PP=4, EP=8, DP=2` on 128 GPUs means each data replica contains a model-parallel submesh of `2 x 4 x 8 = 64` ranks, and there are two replicas that synchronize the trainable state they both own. Increasing one model-parallel dimension usually shrinks the available data-parallel dimension unless the world size also grows.

Training systems

Where collectives sit in modern distributed training

Different parallelism strategies choose different collectives because they choose different tensor ownership. The fastest training job is usually the one that reduces full replication, overlaps communication with compute, and avoids forcing every rank to wait for unrelated data.

Data parallel / DDP

Each rank owns a full model replica and a different microbatch. Backward produces local gradients, then DDP all-reduces gradient buckets so every replica applies the same optimizer update.

FSDP and ZeRO

Parameters, gradients, and optimizer state are sharded. Forward commonly all-gathers parameter shards; backward uses reduce-scatter so each rank keeps only the reduced shard it owns.

Tensor parallel

Layers are split across ranks. Column- and row-parallel linear layers use all-gather, reduce-scatter, or all-reduce depending on whether activations or partial sums are partitioned.

Pipeline parallel

Consecutive layer blocks live on different ranks. The dominant communication is point-to-point activation and gradient transfer, with collectives still used inside data/tensor-parallel groups.

Mixture of Experts

Tokens are routed to expert-owning ranks, often with all-to-all. Imbalanced routing creates stragglers even when the collective itself is efficient.

Evaluation and metrics

Scalar losses use reduce or all-reduce; predictions might use gather or all-gather. These are usually small messages, so launch latency matters more than peak bandwidth.

Performance model

How to reason about scaling

Latency and bandwidth

A simple model is `time ≈ α * steps + bytes / bandwidth`, where `α` is per-step latency. Rings often minimize bytes per link for large tensors, while trees often reduce step count for small tensors.

Chunking and channels

Large tensors are split into chunks and sometimes channels. This lets independent paths carry different pieces, but too many chunks can add scheduling overhead.

Overlap

DDP launches all-reduce as soon as a gradient bucket is ready. Good overlap hides communication behind remaining backward compute; poor bucket order exposes communication on the critical path.

Rank mapping

Logical neighbors should align with physical neighbors. A ring that jumps across sockets or NIC rails can waste bandwidth even when the collective algorithm is theoretically optimal.

Precision and reduction semantics

Reductions are not bitwise associative in floating point. Changing rank order, algorithm, bucket size, or datatype can change low bits. Mixed-precision training must choose accumulation and communication datatypes intentionally.

Stragglers

Collectives synchronize progress. A slow data loader, uneven MoE routing, thermal throttling, or one congested NIC makes fast ranks wait at the next collective.

Debugging playbook

When collectives hang or underperform

Correctness first

  • Every rank must enter the same collective in the same order.
  • Tensor counts and datatypes must match the collective contract.
  • Root arguments are ranks, not CUDA device IDs.
  • Async work must be waited on before consuming results.

NCCL visibility

  • Use `NCCL_DEBUG=INFO` for algorithm, transport, and topology clues.
  • Add `NCCL_DEBUG_SUBSYS=INIT,GRAPH,COLL,NET` when narrowing startup or network issues.
  • Check whether NCCL selected sockets, IB/RDMA, NVLink, or unexpected fallback paths.
  • Inspect rank-to-GPU and rank-to-NIC placement before tuning thresholds.

Performance checks

  • Compare single-node and multi-node bandwidth separately.
  • Run `all_reduce_perf` from nccl-tests for raw collective bandwidth.
  • Profile whether communication is overlapped or on the critical path.
  • Watch for host-side launch gaps, CPU affinity issues, or data-loader stalls.
